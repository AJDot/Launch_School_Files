Writing Your Best Code

This chapter will focus on the best coding practices for HTMl and CSS. This is about the overarching framework for writing HTML and CSS.

HTML Coding Practices

Best coding practices keep code lean and well-organized. We need to write well-structured and standards-compliant markup.

Write Standards-Compliant Markup

Unfortunately, HTML allows poor code to execute and render with varying accuracy. And even when it renders correctly, it doesn't mean it is semantically correct or compliant. Some things to watch keep straight:

  • nest and close all elements correctly
  • use IDs and classes appropriately
  • always validate your code

Bad Code

<p id="intro">New items on the menu today include <strong>caramel apple cider and breakfast crepes</p>.</strong>
<p id="intro">The caramel apple cider is delicious.

Good Code

<p class="intro">New items on the menu today include <strong>caramel apple cider and breakfast crepes</strong>.</p>
<p class="intro">The caramel apple cider is delicious.</p>

Make Use of Semantic Elements

With well over 100 elements, it's tough to know which element is the right one to use to make your code semantically accurate. Need to research and double-check code to ensure we're using the right elements. This will make our website more accessible... and arguably easier to style. If you aren't sure if you're doing it right, recruit a friend to do a code review.

Bad Code

<span class="heading"><strong>Welcome Back</span></strong>
<br><br>
It has been a while. What have you been up to lately?
<br><br>

Good Code

<h1>Welcome Back</h1>
<p>It has been a while. What have you been up to lately?</p>

Use the Proper Document Structure

HTML is forgiving and will render without the use of the <DOCTYPE html> doctype or <html>, <head>, and <body>. But without them we can't be sure exactly how the page will render in every browser.

We must ALWAYS include these elements. This keeps code fully semantic and compliant, leaving code to render predictably.

Bad Code

<html>
  <h1>Hello World</h1>
  <p>This is a web page.</p>
</html>

Good Code

<!DOCTYPE html>
<html>
  <head>
    <title>Hello World</title>
  </head>
  <body>
    <h1>Hello World</h1>
    <p>This is a web page.</p>
  </body>
</html>

Keep the Syntax Organized

What to do what to do when thish task gets large. Here are some rules to follow:

  • Use lowercase letters within element names, attributes, and values
  • Indent nested elements
  • Strictly use double quotes, not single or completely omitted quotes
  • Remove the forward slash at the end of self-closing elements
  • Omit the values on Boolean attributes

<Aside>
<h3>Chicago</h3>
<H5 HIDDEN='HIDDEN'>City in Illinois</H5>
<img src=chicago.jpg alt="Chicago, the third most populous city in the United States" />
<ul>
<li>234 square miles</li>
<li>2.715 million residents</li>
</ul>
</ASIDE>

Good Code

<aside>
  <h3>Chicago</h3>
  <h5 hidden>City in Illinois</h5>
  <img src="chicago.jpg" alt="Chicago, the third most populous city in the United States">
  <ul>
    <li>234 square miles</li>
    <li>2.715 million residents</li>
  </ul>
</aside>

Use Practical ID & Class Values

This is actually a difficult task when writing HTML. The values need to be practical and relate to the content itself (not the style of the content)

Bad Code

<p class="red">Error! Please try again.</p>

Good Code

<p class="alert">Error! Please try again.</p>

Use the Alternative Text Attribute on Images

ALWAYS include the alt attribute on images. Screen readers and other software rely on it completely for image context. It should be very descriptive of what the image contains.

Even if the image isn't relevant the alt should still be included and the value should be left blank so that screen readers will ignore it rather than read the name of the image file..

If an image doesn't have meaningful value (part of the interface), then it should be included as a CSS background image if possible, not as an <img> element.

Bad Code

<img src="puppy.jpg">

Good Code

<img src="puppy.jpg" alt="A beautiful, two-year-old hound mix puppy">

Separate Content from Style

NEVER, EVER< usu inline styles within HTML. These pages take longer to load, are very difficult to maintain, and cause headaches for designer and developers. Instead, use external style sheets.

Avoid a Case of "Divitis"

It's easy to just keep using div for every element but this will bloat the code. After a while, you won't know what each div actually does. To keep code lean and reduce markup, tie multiple styles to a single element whenever possible. And use HTML5 structural elements instead of divs when it is semantically appropriate to do so.

Bad Code

<div class="container">
  <div class="article">
    <div class="headline">Headlines Across the World</div>
  </div>
</div>

Good Code

<div class="container">
  <article>
    <h1>Headlines Across the World</h1>
  </article>
</div>

Continually Refactor Code

Over time website and code bases evolve and grow. Be diligent and remove old/outdated code and styles when possible. AFter you write code, reevaluate it and look for ways to condense it and make it more manageable.

CSS Coding Practices

The goal here is the goal for HTML, lean and well-organized code.

Organize Code with Comments

Bad Code

header { ... }
article { ... }
.btn { ... }

Good Code

/* Primary header */
header { ... }

/* Featured article */
article { ... }

/* Buttons */
.btn { ... }

Write CSS Using Multiple Lines & Spaces

Bad Code

a,.btn{background:#aaa;color:#f60;font-size:18px;padding:6px;}

Good Code

a,
.btn {
background: #aaa;
color: #f60;
font-size: 18px;
padding: 6px;
}

Comments & Spacing

These are two recommendation that should be taken to heart when programming in any language, not just CSS and HTML. The more complex our code becomes the more we need comments to understand/remember what the code is doing. In this way, others will be able to work on your code and not be completely lost. The "other" person could be you 6 months from now.

Use Proper Class Names

Class name (or values) should be modular and should pertain to content with an element, not appearance. They should be written in a way that resembles the CSS language. Use a class like ".alert-message" instead of "Red_Box".

Build Proficient Selectors

CSS selector can get out of control like divs in HTML. They can easily become too long and too location-specific.

The longer a selector is and the more prequalifiers it includes, the higher the specificity it will contain. And higher specificity meeans more likely a slector is to break the CSS cascade and cause undesirable issues.

DO NOT use IDs within selectors. IDs are already as specific as it gets.

Use shorter and primarily direct selectors. Nest them only two to three levels deep, and remove as many location-based qualifying selectors as possible.

Bad Code

#aside #featured ul.news li a { ... }
#aside #featured ul.news li a em.special { ... }

Good Code

.news a { ... }
.news .special { ... }

Use Specific Classes When Necessary

When a CSS selector so long and specific that it no longer make sense then use a class name to sum it all up. This will create more HTMl code but will actually allow faster rendering of the page.

Use Shorthand Properties & Values

It is simply to quickly setup and identify styles. Keeps code minimal as well.

Use Shorthand Hexadecimal Color Values

When available, use the three-charactor shorthand color values andn ALWAYS use lowercase characters with any color value.

Drop Units from Zero Values

A value of 0 is unitless anyway, units don't belong. Zero is always zero. This is eliminate some unnecessary code as well.

Group & Align Vendor Preixes

Bad Code

div {
  background: -webkit-linear-gradient(#a1d3b0, #f6f1d3);
  background: -moz-linear-gradient(#a1d3b0, #f6f1d3);
  background: linear-gradient(#a1d3b0, #f6f1d3);
  -webkit-box-sizing: border-box;
  -moz-box-sizing: border-box;
  box-sizing: border-box;
}

Good Code

div {
background: -webkit-linear-gradient(#a1d3b0, #f6f1d3);
background:    -moz-linear-gradient(#a1d3b0, #f6f1d3);
background:         linear-gradient(#a1d3b0, #f6f1d3);
-webkit-box-sizing: border-box;
   -moz-box-sizing: border-box;
        box-sizing: border-box;
}

Modularize Styles for Reuse

CSS is built to do this. Styles are all about specificity. If a section of news is presented in a box, use a .news class, if there is a featured article, maybe use a .feat-article class.

Summary

Always do these things:

  • Be well-organized, so that it is easy to read, edit, and maintain
  • Be modular and flexible, allowing us to reuse code and patterns as necessary
  • Look as if one person wrote it, even if several people contributed
